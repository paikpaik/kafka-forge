# 프로젝트 계획: Kafka 코어 모듈 (가칭 kafka-core-module)

## 포지셔닝

실제 Kafka(로컬은 Redpanda로 가볍게 운영)를 브로커로 그대로 쓰되, 그 위에 **여러 서비스가 표준화된 방식으로 Kafka를 쓰게 해주는 내부 공유 모듈**을 만든다. `node-core-module`/`node-server-module`이 인증·요청 파이프라인·로깅을 여러 서버가 공유하도록 표준화했던 것과 똑같은 패턴을, 이번엔 이벤트 프로듀스/컨슈밍 영역에 적용한다.

이 방향의 장점:
- **진짜 Kafka를 만진다**: 실제 컨슈머 그룹 리밸런싱, 파티션, 리텐션, replication factor, ISR 같은 개념을 실제 클러스터에서 관찰하며 배운다. "직접 구현"이 아니라 "실제 운영 경험"이 목적이므로 원래 목표(카프카를 경험해보고 싶다)에 정확히 맞는다.
- **회사 실무와 동일한 사고방식으로 확장**: "실제 회사라면 Kafka를 어떻게 모듈화해서 여러 팀이 쓰게 할까"라는 질문 자체가 프로젝트의 핵심 설계 문제가 된다.
- **지금까지 이력서 강점과 자연스럽게 연결**: 원자적 연산/멱등성(중복 적립 방지 경험), 옵저버빌리티(OTel/Prometheus), 정합성(Outbox 패턴)을 전부 재사용할 수 있다.

## 무엇을 만드는가

### 1) 인프라 레이어
- 로컬 Kafka 클러스터 (Redpanda, Docker Compose 한 줄로 기동 — Zookeeper 없이 간단)
- 토픽/파티션/리텐션 설계를 직접 해보고, 파티션 수를 바꿔가며 처리량이 어떻게 달라지는지 실측

### 2) 코어 모듈 (핵심 산출물)
`node-core-module` 스타일의 npm 패키지로 설계. 여러 서비스가 이 모듈만 설치하면 아래를 공짜로 얻는다:

| 기능 | 설명 |
|------|------|
| 표준 Producer | 토픽 네이밍 컨벤션 강제, 메시지 스키마 검증(Zod/JSON Schema), 파티션 키 전략을 옵션화 |
| 표준 Consumer | 컨슈머 그룹 네이밍 컨벤션, 재시도(backoff) + Dead Letter Topic 자동 처리 |
| 멱등성 헬퍼 | 메시지 처리 중복 방지 — 지금까지 Redis 분산락/원자 연산으로 풀던 문제를 Kafka 컨슈머 관점에서 재구현 (예: 처리 완료 오프셋을 별도로 추적해 재시도 시 중복 실행 방지) |
| Outbox 패턴 헬퍼 | DB 트랜잭션 커밋과 Kafka 발행 사이의 정합성 문제 해결 — DB에 먼저 이벤트를 쓰고 별도 프로세스가 폴링해서 Kafka로 발행하는 패턴을 모듈화 |
| 옵저버빌리티 내장 | Produce/Consume마다 OTel span 자동 생성, 컨슈머 랙·처리량·에러율을 Prometheus 메트릭으로 자동 노출 — 별도 설정 없이 모듈 설치만으로 얻어지는 것이 핵심 |
| 안전한 종료 | graceful shutdown 시 진행 중인 메시지 처리를 끝내고 컨슈머를 정상 탈퇴시켜 불필요한 리밸런싱 방지 |

### 3) 레퍼런스 서비스 (사용 예시, 스코프 최소화)
- 이벤트를 발행하는 서비스 1개 + 소비하는 서비스 1~2개로 최소 구성 (예: 주문/알림처럼 단순한 도메인 — 도메인 로직에 시간 쓰지 않기)
- 목적은 "이 모듈을 실제로 여러 서비스에 설치했을 때 정말 표준화가 되는지" 증명

## 단계별 로드맵

### Phase 1 — 로컬 Kafka 환경 + 기본 Produce/Consume
- Redpanda(or Kafka) Docker Compose 기동
- KafkaJS로 raw produce/consume 먼저 찍어보고 동작 원리 체감 (여기서 실제 카프카 개념을 몸으로 익힘)

### Phase 2 — 코어 모듈 v0: Producer/Consumer 래핑
- 토픽 네이밍 컨벤션, 스키마 검증, 파티션 키 전략을 모듈로 추출
- 레퍼런스 서비스 2개에 실제로 설치해서 사용

### Phase 3 — 신뢰성 기능
- 재시도 + DLQ
- 멱등성 헬퍼 (중복 처리 방지)
- graceful shutdown

### Phase 4 — Outbox 패턴
- DB 트랜잭션 + Kafka 발행의 정합성 문제를 실제로 재현하고, Outbox 패턴으로 해결
- 이 프로젝트에서 가장 "회사 실무스러운" 이야기가 될 파트

### Phase 5 — 옵저버빌리티
- OTel 트레이싱 자동 계측 (produce span → consume span 연결, 분산 트레이싱으로 이벤트 흐름 추적)
- Prometheus 메트릭(컨슈머 랙, 처리량, 에러율) + Grafana 대시보드
- 파티션 수를 늘렸을 때 랙/처리량이 실제로 어떻게 바뀌는지 실측 데이터로 보여주기

## 명시적으로 안 하는 것 (v1 스코프 아웃)
- 멀티 브로커 클러스터 직접 운영/튜닝 (단일 노드 Redpanda로 충분, 클러스터링 자체는 관리형 서비스의 몫으로 문서에만 기록)
- Kafka Streams/ksqlDB 같은 스트림 처리 엔진 도입 (코어 모듈의 스코프 밖)
- 완전한 Exactly-once 시맨틱 (트레이드오프 이해 선에서 정리)

## 기술 스택
- Redpanda(로컬 Kafka 호환 브로커, Docker Compose) 또는 실제 Kafka
- TypeScript + Node.js, KafkaJS
- Zod 또는 JSON Schema (메시지 스키마 검증)
- OpenTelemetry + Prometheus + Grafana
- Jest, k6

## 레포 이름 후보
회사 내부 모듈 컨벤션(node-core-module, node-server-module)과 맞추면:
- `kafka-core-module`
- `event-core-module`

독립 오픈소스 브랜드로 갈 경우:
- `streamcore`
- `kafka-toolkit`

## 완료 후 이력서에 담을 수 있는 스토리 (참고)
- "왜 코어 모듈로 표준화했나" — 서비스마다 제각각 Kafka 클라이언트를 쓰면 생기는 문제(네이밍 불일치, 재시도 로직 중복, 관측 불가)를 실제로 겪어보고 모듈화로 해결
- "Outbox 패턴이 왜 필요한가" — DB 커밋과 이벤트 발행 사이의 원자성 문제를 직접 재현하고 해결
- "파티션 수와 컨슈머 랙의 관계" — 실측 데이터 기반으로 스케일 트레이드오프 설명
- "옵저버빌리티를 모듈에 내장한 이유" — 개별 서비스가 매번 계측 코드를 짜지 않아도 되게 만든 설계 판단
